Dockerfile / Build Strategies

You've got a developer repo with a Dockerfile. We want repeatable builds, promotion across dev→stage→prod, and we're in a semi-locked-down environment (internal registry, limited outbound). What deployment/build approach do you recommend in OpenShift and why?

Covers a handful of deployment approaches depending on where you're starting from.

You have a Docker container and want to deploy

Pull image as a deployment and create service

oc new-app docker.io/emccumbe/testapp --name=testapp

Create a route so you can hit the app

oc expose svc testapp
Note: OpenShift runs containers as non-root by default. If the container requires root, it won't work. You'll also want a pull secret for Docker Hub, or set this up via YAML.

Deploy as an ImageStream

An ImageStream tracks images/tags, whether built internally (BuildConfigs) or imported externally (DockerHub). Deployments can point at myapp:latest (ImageStreamTag) instead of hardcoding a registry path — so when a tag updates, the Deployment can pick it up automatically.

Create a Docker Hub pull secret

oc create secret docker-registry dockerhub-cred \ --docker-server=https://index.docker.io/v1/ \ --docker-username=USERNAME \ --docker-password='PASSWORD'

Link secret to service accounts

oc secrets link default dockerhub-cred --for=pull oc secrets link builder dockerhub-cred --for=pull

Import the image as an ImageStream

oc import-image myapp:latest -n NAMESPACE --from=docker.io/emccumbe/myapp:latest --confirm

Deploy from the ImageStream and expose

oc new-app -n NAMESPACE --name myapp --image-stream=myapp:latest -e PORT=8080 oc expose svc/myapp --port=8080 --target-port=8080

You have a Git Repo — let OpenShift build it (S2I / BuildConfig)

A BuildConfig is a Source-to-Image (S2I) feature. OpenShift pulls your Git repo, layers source code onto a base builder image (Python, Node.js, PHP, etc.), and produces a runnable image tracked as an ImageStream.

Create BuildConfig, ImageStream, Deployment, and Service in one shot

oc new-app -n NAMESPACE --name=my-app \ --docker-image=registry.redhat.io/ubi9/php-82:latest \ --code=https://github.com/airick10/democode.git
Note: Make sure you have a Red Hat registry or Docker Hub secret. Public Git repos don't require a secret.

CrashLoopBackOff Triage

A pod is in CrashLoopBackOff. Walk me through your troubleshooting flow in the first 10 minutes. Assume you don't have cluster-admin.

You don't necessarily need cluster-admin for many of these, but it helps.

Check node placement and what's running

oc get pods -o wide

Look for events, image pull failures, mount errors, OOMKilled, probe failures

oc describe pod <PODNAME>

Check logs — use --previous for crash loops, --tail=20 for last 20 lines

oc logs <PODNAME> [--previous] [--tail=20]

Creates a replica pod for troubleshooting — does NOT affect the original

oc debug pod/<PODNAME>

View YAML — good for resource limits, SCCs, etc.

oc get pod <PODNAME> -o yaml

Other Things to Check

  • Check Probes (liveness/readiness)
  • Check Environment Variables and ConfigMaps
  • Confirm the Pull Secret is correct
  • Validate the image name and tag
  • Check resource limits — compare pod vs node
  • Is there a dependent pod/service that is down?

CrashLoopBackOff vs ImagePullBackOff

A CrashLoopBackOff means the container starts, then dies immediately. An ImagePullBackOff means the container never started at all — wrong image name, bad pull secret, etc.

If logs show nothing because it exits immediately, what next?

Look at oc describe — check Events, Last State, Reason, Exit Code. Look for OOMKilled, probe failures, mounting errors.

Last 30 events in namespace, sorted by time

oc get events -n <NAMESPACE> --sort-by=.lastTimestamp | tail -30

Large File Transfer / No rsync or tar

Rsync isn't available and the container image is minimal (no tar, no shell utilities). You need to get a 2GB file into the running workload. What are your options and tradeoffs?

Normally you can transfer files from your bastion/gateway server fairly easily.

Standard rsync (when available)

oc rsync ./config/ <Namespace>/<Podname>:/opt/app/config/

oc cp — uses tar under the hood, so won't work on minimal images

oc cp ./config/ <Namespace>/<Podname>:/opt/app/config/

Option: Helper Pod Sharing a PVC

Create a minimal "helper" pod that mounts the same PVC your target pod uses. Copy files there — your main pod sees them via the shared PVC.

apiVersion: v1 kind: Pod metadata: name: pvc-helper spec: containers: - name: helper image: registry.access.redhat.com/ubi9/ubi-minimal command: ["/bin/sh", "-c", "sleep infinity"] volumeMounts: - name: shared-data mountPath: /data volumes: - name: shared-data persistentVolumeClaim: claimName: my-pvc

Apply and copy to helper pod

oc apply -f pvc-helper.yaml oc cp ./myfile.txt <Namespace>/pvc-helper:/data/myfile.txt
Note: If your real app pod mounts the same PVC, the file will be visible there — potentially at a different mount path.

Option: ConfigMap (Small Files Only)

Works well for config files. Not suitable for a 2GB file — but good to know.

Create ConfigMap from file

oc create configmap my-file-cm --from-file=myfile.txt
apiVersion: v1 kind: Pod metadata: name: test-pod spec: containers: - name: app image: registry.access.redhat.com/ubi9/ubi-minimal command: ["/bin/sh", "-c", "sleep infinity"] volumeMounts: - name: filevol mountPath: /incoming readOnly: true volumes: - name: filevol configMap: name: my-file-cm

File will be visible at /incoming/myfile.txt inside the pod.

Note: This YAML mounts the ConfigMap as a read-only volume. There's no init container here — the file is just directly available inside the running container at the mount path. If you need to copy it somewhere writable first, you'd add an init container to do that step.

Machine Config Operator Degraded

Machine Config Operator is Degraded and one MachineConfigPool is not updating. Walk me through how you would triage this and what your priorities are.

During upgrades, MCO updates last and goes through each node — so seeing it degraded during that process is normal. It takes the most time.

List all Cluster Operators — check Available and Degraded columns

oc get co

Describe the machine-config operator — look at degraded reasons and messages

oc describe co machine-config

Check MachineConfigPool — UPDATED, UPDATING, DEGRADED, machine count vs ready

oc get mcp

Which nodes are NotReady, SchedulingDisabled, or Cordoned?

oc get nodes

MCO logs — the MCD runs on each node

oc logs -n openshift-machine-config-operator <POD>
Look for patterns: disk full on a node, OS mismatch on upgrade, stuck draining (DB locks?), update partially applied, node reboot failed.

What NOT to do when MCO is Degraded

  • Don't update manually
  • Don't manually SSH and modify OS state
  • Don't force a reboot
  • Don't delete MachineConfigs blindly
  • Don't break the ignition state

Platform-Level Security Controls

What platform-level controls do you implement to secure an OpenShift cluster?
  • SCCs (Security Context Constraints)
    oc get pod <PODNAME> -n <NAMESPACE> -o jsonpath='{.metadata.annotations.openshift\.io/scc}{"\n"}'
    oc get scc
    oc adm policy who-can use scc privileged
  • Avoid Privileged Containers
    oc get pod <PODNAME> -o yaml | grep -A5 securityContext

    Look for: privileged: true, runAsUser: 0, allowPrivilegeEscalation: true

  • NetworkPolicies
    oc get networkpolicy -A
    oc describe networkpolicy <NAME> -n <NAMESPACE>
  • RBAC Least Privilege
    oc adm policy who-can create pods oc get rolebinding -A oc get clusterrolebinding oc auth can-i create pods --as=<USER> -n <NAMESPACE>
  • Image Scanning / Trusted Registries — Don't pull random Docker Hub images in prod
    oc get pods -A -o jsonpath='{range .items[*]}{.spec.containers[*].image}{"\n"}{end}'
    oc get imagecontentsourcepolicy
  • Pull Secrets
    oc get secrets -n <NAMESPACE> oc get sa default -o yaml -n <NAMESPACE>
  • Audit Logs
    oc adm node-logs <MASTERNODE> --path=kube-apiserver/audit.log
    oc get apiserver cluster -o yaml | grep audit
  • OIDC Integration — Don't use htpasswd in production
    oc get oauth cluster -o yaml oc get users oc get identities
  • Secrets Management — Mount as files or use external stores, not plain env vars
    oc get pods -A -o yaml | grep -i env
    oc get secrets -A
  • Separate Infrastructure Nodes for ingress/monitoring (if needed)
    oc get nodes --show-labels | grep node-role oc get pods -n openshift-ingress -o wide oc get pods -n openshift-monitoring -o wide
OpenShift does not run pods/containers as root by default — a key distinction from vanilla Kubernetes. All pods default to the Restricted SCC. More on SCCs → | Pod Security Policies →

How would you prevent developers from running privileged containers?

  • Use the restricted SCC
  • RBAC restrictions
  • Admission controls
  • Don't grant anyuid or privileged
  • Review service accounts
  • Namespace-level controls

What's your approach to secrets management?

  • Avoid plain environment variables in Git
  • External secret stores (Vault, AWS Secrets Manager, etc.)
  • Sealed Secrets or External Secrets Operator
  • RBAC on secret access
  • Secret rotation strategy — regularly rotating passwords, API keys, and encryption keys on a schedule

Minor Version Upgrade

How do you plan and execute an OpenShift minor version upgrade in production?

Upgrade command (from internal registry — brute force but functional)

oc adm upgrade --allow-explicit-upgrade --force \ --allow-upgrade-with-warnings \ --to-image internalserver.domain.com/openshift/release:4.18.30-x86_64

Pre-Upgrade Checklist

  • Check release notes / breaking changes
  • Validate all cluster operators are healthy — oc get co
  • Upgrade a non-production cluster first
  • Review deprecated APIs
  • Backup etcd →
  • Maintenance window planning
  • Understand node draining behavior
  • Plan post-upgrade validation (smoke tests)

Post-Upgrade Validation

  • Deployments — Create a test deployment. Does it schedule and run?
  • Service/Ingress — Verify connectivity via route/ingress
  • Storage — Verify PVs are bound, pods can read/write
  • CLI/API — Run oc commands — do they respond?

One worker node doesn't return after the upgrade. What do you do?

MCP (MachineConfigPool) — defines which nodes are in the pool via labels, tracks status.
MCD (MachineConfigDaemon) — daemonset on each node, applies config, drains, reboots.

  • Check MCP
    oc get mcp oc describe mcp worker

    Look for Updating=True, Degraded=True, MachineCount vs UpdatedMachineCount

  • Check MCD Logs
    oc get pods -n openshift-machine-config-operator -o wide | grep <NODENAME> oc logs -n openshift-machine-config-operator <MCDPOD> -c machine-config-daemon

    Look for drain failures, reboot loops, config apply errors, failed OS pivot

  • Node stuck draining?
    oc describe node <NODENAME>

    Look for pods that won't evict

  • Disk Full?
    oc debug node/<NODENAME> # then: chroot /host df -h df -i

    Disk at 90%+ is a problem. Check for inode exhaustion too.

  • Cordon + Manual Investigation
    oc debug node/<NODENAME> # then: chroot /host systemctl status kubelet journalctl -u kubelet -n 100
    oc adm drain <NODE> --force --ignore-daemonsets --delete-emptydir-data
  • Check cloud provider console if IaaS
  • Don't immediately delete the node unless absolutely necessary

Cluster Performance / Slowness

Applications are reporting slowness, but all ClusterOperators are Available and no alerts are firing. How do you approach this?

Finding the bottleneck when everything looks "fine" on the surface.

  • API Server Latency
    oc get --raw /metrics | grep apiserver_request_duration_seconds_bucket | head
    oc get --raw /readyz?verbose
  • etcd fsync Latency
    oc -n openshift-etcd exec -it <ETCD-PODNAME> -- \ curl -s https://localhost:2381/metrics | grep fsync

    Look for etcd_disk_wal_fsync_duration_seconds — anything at 10ms or above is concerning

  • Node CPU Steal / Throttling
    oc adm top nodes oc debug node/<NODENAME> # chroot /host, then: top
  • Disk I/O Pressure
    oc debug node/<NODENAME> # chroot /host, then: iostat -x 1
    oc describe node <NODENAME> | grep -i diskpressure

    await > 20ms is bad. %util near 100% = saturated.

  • Network Congestion
    oc exec <PODNAME> -- netstat -s | grep retrans
    oc debug node/<NODENAME> # chroot /host, then: sar -n DEV 1

    Looking for high RX/TX errors and packet drops

  • DNS Resolution Issues
    oc exec -it <PODNAME> -- nslookup kubernetes.default
    oc -n openshift-dns logs -l dns.operator.openshift.io/daemonset-dns=default
  • Router Saturation (Ingress)
    oc -n openshift-ingress exec deploy/router-default -- \ curl -s localhost:1936/metrics | grep haproxy
    oc adm top pods -n openshift-ingress

    High CPU + latency = bottleneck at the router

  • HPA Thrashing (Horizontal Pod Autoscaler)
    oc get hpa -A oc describe hpa <HPANAME>

    Look for scaling events and oscillating replica counts

  • Pod Restarts Trend
    oc get pods -A --sort-by=.status.containerStatuses[0].restartCount
    watch -n 5 'oc get pods -A | grep -v Running'
What hurts most: etcd latency and disk I/O cause the most widespread pain. DNS issues cause "everything feels weird" symptoms. CPU throttling silently kills performance and is often missed.

How do you determine if this is cluster-wide or namespace-specific?

  • Compare metrics across multiple namespaces
  • Check router metrics
  • Review API latency metrics
  • Check node-level saturation
  • Determine if only certain workloads are affected

etcd High Latency & Leader Changes

etcd shows high latency and leader changes. What do you investigate first?
  • etcd fsync Latency
    oc -n openshift-etcd exec -it <ETCD-PODNAME> -- \ curl -s https://localhost:2381/metrics | grep fsync

    etcd_disk_wal_fsync_duration_seconds — 10ms+ is concerning

  • Disk Full / Pressure
    oc debug node/<NODENAME> # chroot /host, then: df -h
    oc describe node <MASTERNODE> | grep -i diskpressure

    85%+ disk can be an issue

  • Resource Starvation (CPU/Memory)
    oc adm top nodes oc adm top pods -n openshift-etcd oc debug node/<MASTERNODE> # chroot /host, then: top
  • Network Latency Between Masters
    oc -n openshift-etcd exec -it <ETCDPOD> -- \ etcdctl endpoint status --cluster -w table
    oc debug node/<MASTERNODE> # chroot /host, then ping another master IP

    Look for LATENCY column and leader changes in the table

  • Excessive API Churn
    oc get events -A --sort-by=.lastTimestamp | tail -50
    oc adm top pods -A --sort-by=cpu

    Look for noisy controllers or users hammering the API

  • Backup Frequency (etcd backups running too often?)
    oc get cronjobs -A oc get pods -n openshift-etcd
  • Quorum / Leader Changes
    oc -n openshift-etcd exec -it <ETCDPOD> -- \ etcdctl endpoint status --cluster -w table
    oc -n openshift-etcd exec -it <ETCDPOD> -- \ etcdctl member list -w table

    Look for leader flipping frequently and inconsistent raft index

Usually: fsync latency and disk I/O are the culprits. Disk or network tend to be the root cause.

Would you reboot a master immediately?

No. Not casually. Understand the quorum first. Do health checks. If a reboot is necessary, do it one node at a time.

Worker Node NotReady

A worker node suddenly goes NotReady. Walk me through how you would investigate and decide what to do next.

This stems from a handful of possibilities — troubleshooting the node itself, understanding kubelet, and cloud vs. on-prem behavior.

Grab node status — look for Ready, SchedulingDisabled, NotReady

oc get nodes

Look at node details — MemoryPressure, DiskPressure, PIDPressure, NetworkUnavailable, taints, recent events

oc describe node <NODE>

Log into the node — run chroot /host, then journalctl -u kubelet to see kubelet logs

oc debug node/<NODE>

If multiple nodes are behaving oddly, check operators and MCP

oc get co oc get mcp

Things to investigate inside oc debug (chroot /host first)

  • Pod placement on the node
    oc get pods -A -o wide | grep <NODENAME>
  • Suspect it's stuck draining
    oc describe node <NODENAME>
    oc get pdb -A

    A node can look weird if it's stuck draining — often due to a PDB blocking eviction

  • Kubelet logs
    systemctl status kubelet
    journalctl -u kubelet -n 100 --no-pager

    Is kubelet running? Crash looping? Cert errors?

  • Container runtime (CRI-O)
    systemctl status crio
    journalctl -u crio -n 100 --no-pager

    If kubelet is fine but CRI-O is stuck, pods may be dead or node status degraded

  • Restart CRI-O and Kubelet — only if you have a good idea why it failed
    systemctl restart crio systemctl restart kubelet
  • Disk space
    df -h df -i

    Check inodes too — inode exhaustion looks just like full disk

  • Resource pressure
    top free -h
    oc adm top nodes
  • Network connectivity to API
    ping <API-VIP-OR-API-HOST>
    curl -k https://api.<CLUSTERNAME>:6443/readyz

    If the host is alive but can't reach the API, that points to a network issue

Partitioned vs. Dead Node

Partitioned: VM/host is alive, but kubelet or network path is broken.
Dead Node: VM/host is actually down, hung, or unreachable.

If this is draining/upgrade-related

oc describe mcp worker oc get pods -n openshift-machine-config-operator -o wide | grep <NODENAME> oc logs -n openshift-machine-config-operator <MCD-POD> -c machine-config-daemon

Look for stuck draining or PDB blockage. Sometimes it just takes time to drain and configure.

Don't immediately delete the node! It may be in a draining state. Don't restart it without investigating first.

PVC Not Binding / Pod Pending

A pod is stuck in Pending state due to a PVC not binding. How do you troubleshoot this?

Having a strong understanding of PVC and persistent storage is key in OpenShift. Also helpful to understand CSI.

Check pod events — look for FailedScheduling, unbound PersistentVolumeClaims

oc describe pod <PODNAME> -n <NAMESPACE>

List PVCs and check bind status — look for Pending in the STATUS column

oc get pvc -n <NAMESPACE>

Describe the PVC — events at the bottom are key

oc describe pvc <PVCNAME> -n <NAMESPACE>

Things to dig into

  • StorageClass correctness
    oc get sc | grep '(default)'

    Verify the correct provisioner (ebs.csi, ceph, etc.) and that a default SC exists. Look for a wrong or missing SC.

  • AccessModes mismatch
    oc get pvc <PVC> -o yaml | grep accessModes -A2
    oc get pv

    Common mismatch: PVC requests ReadWriteMany but backend only supports ReadWriteOnce

  • Capacity issues
    oc get pv
    oc describe pvc <PVC>

    Is the PV large enough? Does the requested storage match what's available?

  • Check provisioner pods (rook-ceph, EBS CSI, etc.)
    oc get pods -A | grep -i csi
    oc logs <CSIPOD> -n <NAMESPACE>

    If the provisioner is down, the PVC stays Pending

  • Node affinity / topology constraints
    oc describe pvc <PVC>
    oc describe pod <PODNAME>

    Look for "waiting for first consumer" and node selectors/affinity restricting storage to certain nodes

Dynamic vs. Static Provisioning

Dynamic: PVC → StorageClass → provisioner creates PV automatically. Failures usually mean a broken provisioner, bad SC, or topology mismatch.

Static: PV must exist before a PVC binds. Issues can be no matching PV, wrong size, wrong access mode, wrong StorageClass, or already bound.

Quick test flow

oc describe pvc <PVC> oc get storageclass oc get pv oc get pods -A | grep -i csi oc describe pod <POD>

Route Created, External Access Broken

A route is created, but external users cannot access the application. Internal pod-to-pod traffic works. What do you check?

External access goes through the ingress controller (router). Routes map external traffic → service → pods. If internal works but external doesn't, the problem is almost always the route, router, DNS, or load balancer — not the app.

Check the route — look for correct host, service name, port, and TLS config

oc get route -n <NAMESPACE>
oc describe route <ROUTENAME> -n <NAMESPACE>

Check service and endpoints — endpoints should show pod IPs and ports (e.g. 10.128.2.15:8080)

oc get svc <SERVICENAME> -n <NAMESPACE>
oc get endpoints <SERVICENAME> -n <NAMESPACE>

Empty endpoints = wrong service selector, pods not ready, or wrong port. Route is fine but has nowhere to send traffic.

Check ingress/router pods — running? crashing? CPU pinned?

oc get pods -n openshift-ingress oc get svc -n openshift-ingress oc adm top pods -n openshift-ingress

Other things to check

  • TLS Termination Issues
    oc logs -n openshift-ingress deploy/router-default
    oc get route <ROUTE> -o yaml

    Look for backend connection failures, TLS errors, route not admitted. Check status.ingress.conditions — Admitted should be true. Also check the termination type: Edge, Passthrough, or Reencrypt.

  • DNS Resolution
    nslookup <ROUTEHOST>

    Should resolve to the router's external IP or load balancer. If not, it's a DNS issue.

  • Firewall / Load Balancer outside the cluster
    oc get svc -n openshift-ingress

    Look for LoadBalancer IP (cloud) or NodePort (on-prem). Verify 80/443 is open and LB is pointing to the right nodes.

TLS Termination Types

  • Edge — Router terminates TLS. Backend is plain HTTP.
  • Passthrough — TLS goes directly to the pod. No HTTP inspection.
  • Reencrypt — TLS to router, then TLS again to the pod.

Quick test flow

oc describe route <ROUTE> oc get endpoints <SERVICENAME> oc get pods -n openshift-ingress nslookup <HOST> curl -vk https://<HOST>

Pod Won't Schedule — Resources Available

A developer says their pod won't schedule, but there are nodes with free resources. What could be happening?

The scheduler doesn't just look at free resources. It evaluates constraints like quotas, affinity, taints, and policies. If any constraint fails, the pod won't schedule.

Check pod events — look for FailedScheduling and the reason (0/5 nodes available, taint mismatch, quota exceeded, etc.)

oc describe pod <PODNAME> -n <NAMESPACE>

Check namespace quota — look for CPU/RAM limits exceeded or object count limits like Pods: 10/10

oc get quota -n <NAMESPACE>
oc describe quota -n <NAMESPACE>

Other things to look for

  • LimitRanges
    oc get limitrange -n <NAMESPACE>
    oc describe limitrange -n <NAMESPACE>

    See if default requests are too high or max limits exceeded

  • Node Selectors / Affinity
    oc get pod <PODNAME> -o yaml | grep -A5 nodeSelector
    oc get pod <PODNAME> -o yaml | grep -A10 affinity
    oc get nodes --show-labels

    Selector doesn't match any node? Zone/region mismatch?

  • Taints and Tolerations
    oc describe node <NODE> | grep -i taint
    oc get pod <PODNAME> -o yaml | grep -A5 tolerations

    If a node has a taint and the pod doesn't tolerate it, it won't schedule

  • Resource Requests > Allocatable
    oc describe node <NODE>
    oc adm top nodes

    Look at Allocatable vs. Allocated Resources. CPU/memory fragmentation can be an issue even when total looks fine.

  • PVC / Storage Constraints
    oc describe pvc -n <NAMESPACE>

    If you see "pod has unbound immediate PVC," the scheduler won't place the pod

  • SCC / Security Constraints
    oc describe pod <PODNAME> | grep -i scc
    oc get events -n <NAMESPACE>

    Sometimes the pod can't be admitted due to SCC violations

Note: Scheduler decisions are based on requests, not actual usage. A node can show low utilization but still be "full" from the scheduler's perspective. Also check namespace-level constraints.
